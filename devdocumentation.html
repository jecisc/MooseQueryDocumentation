Developer Documentation

Last updated: Jan 23th, 2018

Introduction

MooseQuery is the result of the Inria research team RMoD. This page documents the internals of MooseQuery. As prerequisites, it supposes the reader has a minimum of knowledge of Fame and Famix. If not, please read the Moose documentation. if you wish to understand everything.

This documentation covers:

  • How to complete a Famix meta-model to make new entities/relations queryable via MooseQuery
  • How the containment tree exploration described in the user documentation works
  • How the navigation queries works
  • How the scope queries works

Completion of the Meta-model

The content of this section will cover the part of new entities/relations creation specific to MooseQuery. It will not cover how to create new Moose entities. For this, please refer to Moose documentation.

First, the developer of the model have to make his new entities/relations queryable with MooseQuery.

Relations definitions

As said in the user documentation, MooseQuery is based on two concepts:

  • Imbrication of entities: which entity contains which entities.
  • Associations between the entities: how the entities interact with each other.

Here, for example, Entity1 is the container of Entity2 and also Entity1 is the source of an association to Entity3.

Moose query needs to know those relations in order to work. To declare them you just need to use three pragma: <container>, <source> and <target>

Containment Relation Definition

FAMIXAttribute>>parentType
	<MSEProperty: #parentType type: #FAMIXType opposite: #attributes>
	<MSEComment: 'Type declaring the attribute. belongsTo implementation'>
	<container> "<=== Here is the containment relation definition"
	^ parentType
Association Definition

FAMIXAccess>>accessor
	<MSEProperty: #accessor type: #FAMIXBehaviouralEntity opposite: #accesses>
	<MSEComment: 'Behavioural entity making the access to the variable. from-side of the association'>
	<source> "<=== Here is the association's source definition"
	^ accessor

FAMIXAccess>>variable
	<MSEProperty: #variable type: #FAMIXStructuralEntity opposite: #incomingAccesses>
	<MSEComment: 'Variable accessed. to-side of the association'>
	<target> "<=== Here is the association's target definition"
	^ variable

Traits usage

The definition of the relations give to MooseQuery all the requirements for querying the meta-model. Now, the new entities of the model need to have the behaviour, i.e., the methods, to execute the queries. To do so, MooseQuery comes with a set of Traits to use in the new entities.

Trait Default users Description
TAssociationMetaLevelDependency FAMIXAssociation This trait includes the information about associations needed for the navigation queries. It needs to be used by every association in a meta-model.
TEntityMetaLevelDependency FAMIXAnnotationInstance, FAMIXAnnotationInstanceAttribute, FAMIXNamedEntity This trait includes the information about entities and adds the behaviour to explore the containment tree and execute scope queries. It needs to be used by every entity in a meta-model.
TDependencyQuery FAMIXAnnotationInstance, FAMIXAnnotationInstanceAttribute This trait includes the information about entities and adds the behaviour to execute navigation queries. It needs to be used by every entity in a meta-model.
TOODependencyQuery FAMIXNamedEntity This trait includes the information about object-oriented entities and adds the behaviour to execute navigation queries. It needs to be used by every object-oriented entities in a meta-model instead of TDependencyQuery.

All those traits need to be used when implementing a new meta-model. For performance reasons, some more work is needed when using those traits. Some methods exist in two versions. One private with the real behaviour, and the other explicitly required to get a cached version. Here is the list of caches to implement:

  • TAssociationMetaLevelDependency class
    • #sourceTypes
    • #targetTypes
  • TEntityMetaLevelDependency class
    • #allChildrenTypes
    • #allParentTypes
    • #childrenSelector
    • #parentSelector
    • #allIncomingAssociationTypes
    • #allOutgoingAssociationTypes
    • #incomingMSEProperties
    • #outgoingMSEPreperties
Example of Cache Implementation

TEntityMetaLevelDependency class>>allChildrenTypes
	^ self explicitRequirement

TEntityMetaLevelDependency class>>privateAllChildrenTypes
	^ (self childrenTypes withDeepCollect: #childrenTypes as: Set) asOrderedCollection

FAMIXNamedEntity class>>allChildrenTypes
	^ allChildrenTypes ifNil: [ allChildrenTypes := self privateAllChildrenTypes ]

Containment Tree Exploration

Now that we saw how to make entities queryable, we begin to check the internal mechanisms of MooseQuery. The easiest queries are the ones related to the exploration of the containment tree.

MooseQuery allows querying the children/parents of an entity. This behaviour is based on the meta-model of the application. Each Moose's entity possesses a meta-description that lists all the properties of the entity. These properties can be obtained with:

Getting the Properties of an Entity

MooseEntity class>>allDeclaredProperties
	"All properties described in the metamodel"
	^self mooseDescription allAttributes


FAMIXType allDeclaredProperties. "==> An array of property descriptions"

Each property knows:

  • If it is a property defining the containment of other entities (via the "container" pragma)
  • If it is a property defining a parent entity (via the "container" pragma)
  • The selector returning the property in the entity's API

So, it is possible to get the selectors of an entity returning the potential parents or children.

Getting the parents/children selectors of an entity

TEntityMetaLevelDependency class>>privateChildrenSelectors
	^ self allDeclaredProperties select: #isChildrenProperty thenCollect: #implementingSelector

TEntityMetaLevelDependency class>>privateParentSelectors
	^ self allDeclaredProperties select: #isContainer thenCollect: #implementingSelector

FAMIXType parentSelectors. "=> #(#container #parentPackage)"
FAMIXType childrenSelectors.  "=> #(#annotationInstances #types #definedAnnotationTypes #attributes #functions #methods)"

Once we retrieved all parents/children selectors it is possible to perform them on an entity to get their results and gather the children/parents of this entity.

Getting the parents/children of an entity

TEntityMetaLevelDependency>>children
	| res |
	res := OrderedCollection new.
    "Since some selector can return nil, an entity or a collection of entities, we need to do a nil check and to ensure everything is a collection via #asCollection"
	self childrenSelectors do: [ :accessor | (self perform: accessor) ifNotNil: [ :r | res addAll: r asCollection ] ].
	^ res asSet


TEntityMetaLevelDependency>>parents
	| res |
	res := OrderedCollection new.
    "Since some selector can return nil, an entity or a collection of entities, we need to do a nil check and to ensure everything is a collection via #asCollection"
	self parentSelectors do: [ :accessor | (self perform: accessor) ifNotNil: [ :r | res addAll: r asCollection ] ].
	^ res asSet

It is now possible to get all the children/parents of an entity via the meta-model. With this, you exclude the problem of human error.

The next step is to be able to get children/parents recursively. Since containment trees can be pretty big, we use an accumulator to avoid creating a lot of collections.

Getting the parents/children of an entity recursively

TEntityMetaLevelDependency>>allChildren
	"Returns all the children and sub-children of an entity, i.e my children and those of my children, and those of the children of my children, etc"
	^ self addAllChildrenIn: OrderedCollection new


TEntityMetaLevelDependency>>addAllChildrenIn: aCollection
	aCollection addAll: self children.
	self children do: [ :each | each addAllChildrenIn: aCollection ].
	^ aCollection

Navigation Queries

This section will explain how the navigation queries works.

Navigation queries, as explained in the user documentation, are parametrized by:

  • The direction of the query. Should the receiver be the source or the target of the associations to gather?
  • The scope of the query. Should the query return the associations of the receiver, or the associations of the receiver and its children.
  • The association(s) to query. It is possible to query a specific association or all the associations.

All those parameters will be explained in this section.


To launch a navigation query, all Moose entities can use an API provided by the trait TDependencyQueries, as explained previously. Those methods will all begin in the same way. They will send the request to a new instance of MooseQueryCalculator parametrized with the direction of the query.

Management of the queries direction

This section will cover the query direction management.

As shown in the UML diagram above, directions are managed via a Design Pattern strategy. A MooseQueryCalculator will refer to a strategy that will be either a MooseQueryIncomingDirectionStartegy or a MooseQueryOutgoingDirectionStrategy. The strategies will manage everything related to the direction inside the query. This includes:

  • Obtaining the possible associations types for a given entity depending on the direction
  • Obtaining the meta-properties of an entity defining a relation of source/target between the entity and an association
  • Creating a MooseQueryResult for the right direction
Example of Responsibility Dispatch Between Strategies

MooseQueryAbstractDirectionStrategy class>>associationTypesFor: anEntity
	^ self subclassResponsibility


MooseQueryIncomingDirectionStrategy class>>associationTypesFor: anEntity
	^ anEntity incomingAssociationTypes


MooseQueryOutgoingDirectionStrategy class>>associationTypesFor: anEntity
	^ anEntity outgoingAssociationTypes

The selection of the right strategy is done via the symbol #in or #out given by the user. The MooseQueryAbstractDirectionStrategy is able to return the right strategy with this symbol as input.

Strategy Selection Mechanism

MooseQueryAbstractDirectionStrategy class>>fromSymbol: aSymbol
	"I return the right strategy from a Symbol representing a direction"
	^ self allSubclasses detect: [ :each | each determinatingSymbol = aSymbol ] ifNone: [ self error: 'The symbol passed to the query should be either #in or #out. Here we got ' , aSymbol asString ]


MooseQueryIncomingDirectionStrategy class>>determinatingSymbol
	^ #in


MooseQueryOutgoingDirectionStrategy class>>determinatingSymbol
	^ #out

Local queries

Now that it is clear how the distinction around the direction of the query is managed, we can see how the navigation queries works. Let's begin with the local queries.

Local navigation queries gather associations related to an entity only. It will not gather associations of its children. It is possible to launch a local query via the selector TDependencyQuery>>#queryLocal:with: taking as parameter the direction of the query and the association class to gather. This method will create a new MooseQueryCalculator parametrized with the right direction strategy and call MooseQueryCalculator>>#queryLocal:for: with the association class and the receiving entity as parameters.

MooseQueryCalculator>>#queryLocal:for: will gather the associations and create a new MooseQueryResult with them, as shown in the script bellow.

MooseQueryCalculator>>#queryLocal:for:

 MooseQueryCalculator>>queryLocal: aFAMIXClassAssociation for: anEntity

	^ self strategy queryResultOn: anEntity with: (self queryLocal: aFAMIXClassAssociation for: anEntity in: OrderedCollection new) asSet

In order to gather the associations, we first check that the entity can have instances of the association class. For this, we ask via the strategy all the association types possible for the entity. If the association is in the list the query will proceed. Else, the query will return an empty result.

In the case the entity can contains instances of the association, the query will ask to the direction strategy the meta-properties returning associations in the right direction, iterate on them and collect the result of the property returning the right association.

MooseQueryCalculator>>#queryLocal:for:in:

 MooseQueryCalculator>>queryLocal: aFAMIXClassAssociation for: anEntity in: aCollection
	"check that aFAMIXClassAssociation is well an incoming association type"

	((self strategy associationTypesFor: anEntity) includes: aFAMIXClassAssociation) ifFalse: [ ^ aCollection ].

	"I know that we could use #detect:ifFound: here, but this algo takes too long on big model, so this is a speed up while the sista vm is not ready for production"
	(self strategy msePropertiesOf: anEntity)
		do: [ :prop |
			prop type implementingClass = aFAMIXClassAssociation
				ifTrue: [ (anEntity perform: prop name)
						ifNotNil: [ :coll |
							aCollection addAll: coll asCollection.
							^ aCollection ] ] ]

The code of this method is more complex than it should be because of performance reasons. This method can be called a lot in certain case and it need to be as efficient as possible. A more readable but less efficient version could be:

MooseQueryCalculator>>#queryLocal:for:in:, readable version

 MooseQueryCalculator>>queryLocal: aFAMIXClassAssociation for: anEntity in: aCollection
	"check that aFAMIXClassAssociation is well an incoming association type"

	((self strategy associationTypesFor: anEntity) includes: aFAMIXClassAssociation) ifFalse: [ ^ aCollection ].

	(self strategy msePropertiesOf: anEntity)
		detect: [ :prop | prop type implementingClass = aFAMIXClassAssociation ]
		ifFound: [ :prop | (anEntity perform: prop name) ifNotNil: [ :coll | aCollection addAll: coll asCollection ] ].
	^ aCollection

Global queries

Local queries are useful, but most of the time users want to get all the associations inside the entity, including in their children.

These queries begin in the same way as local navigation queries. The entity will create a new MooseQueryCalculator parametrized with a direction strategy and return a new MooseQueryResult with the gathered association. The only part changing is the gathering of the associations.

It will begin by gathering the associations of the receiver via the same method as the local navigation queries (MooseQueryCalculator>>queryLocal:for:in:), and then iterate on the children of the entity (which is possible via the methods of the containment tree exploration) to call recursively the query.

MooseQueryCalculator>>#query:for:in:

 MooseQueryCalculator>>query: aFAMIXClassAssociation for: anEntity in: aCollection
	"I know that we could use #do: here, but this algo takes too long on big model, so this is a speed up while the sista vm is not ready for production"

	| selectors |
	self queryLocal: aFAMIXClassAssociation for: anEntity in: aCollection.
	1 to: (selectors := anEntity childrenSelectors) size do: [ :index |
		(anEntity perform: (selectors at: index))
			ifNotNil: [ :children |
				| coll |
				1 to: (coll := children asCollection) size do: [ :i | self query: aFAMIXClassAssociation for: (coll at: i) in: aCollection ] ] ].
	^ aCollection

As for the local queries, the readability of the method is sacrificed for performances. Find bellow a more readable version.

MooseQueryCalculator>>#queryLocal:for:in:, readable version

 MooseQueryCalculator>>query: aFAMIXClassAssociation for: anEntity in: aCollection
	"I know that we could use #do: here, but this algo takes too long on big model, so this is a speed up while the sista vm is not ready for production"

	self queryLocal: aFAMIXClassAssociation for: anEntity in: aCollection.
	anEntity childrenSelectors
		do: [ :selector | (anEntity perform: selector) ifNotNil: [ :children | children asCollection do: [ :child | self query: aFAMIXClassAssociation for: child in: aCollection ] ] ].
	^ aCollection

Queries on all associations

One use of navigation queries is to find the dependencies between entities. To find all the dependencies of an entity it is necessary to gather all associations.

MooseQuery allows to directly gather all the associations incoming or outgoing of an entity at once. The implementation will gather them in one containment tree visit for performance reasons instead of iterating on all associations to execute the queries.

The main change in the implementation is done in the local query. Instead of selecting the meta-property where the implementing type is a specific association class, it will gather the result of all properties defining a incoming/outgoing relation with the entity.

MooseQueryCalculator>>#queryAllLocalFor:in:

 MooseQueryCalculator>>queryAllLocalFor: anEntity in: aCollection
	| properties |
	"Instead of #to:do: we could just use #do: but this implementation is much faster. Maybe sista will remove the needs of the todo later."
	1 to: (properties := self strategy msePropertiesOf: anEntity) size do: [ :ind | (anEntity perform: (properties at: ind) name) ifNotNil: [ :coll | aCollection addAll: coll asCollection ] ].
	^ aCollection

Scope Queries

This section will explain how the scope queries works.

A scope query is parametrized by:

  • The direction of the query.
    • Up in the containment tree
    • Down in the containment tree
    • Both up and down in the containment tree
  • The Famix entity class defining the scope to query

A scope query can be applied on:

  • A Famix entity
  • A MooseQueryResult obtained from a navigation query

Query on a Famix entity

Queries going up or down in the containment tree for the scope search are usable via the methods TEntityMetaLevelDependency>>#atScope: and #toScope:.

Both those methods will begin to create a collection to store the results and call #atScope:in:/#toScope:in: with the newly created collection as parameter. This is done for performance reasons. Instead of creating multiple collection and use concatenation on them, we create on at the beginning and pass it as parameter of the query methods.
